User Guide · 牙科设备推荐Agent

从“数据导入”到“推荐落地”
一份真正能上手的完整使用指南

这套系统不是一个单纯的聊天机器人，也不是一张静态报表。 它是一套围绕 商品主数据、客户购买记录、用户画像、推荐生成、Agent 辅助操作 组成的业务工作台。你可以把它理解为一位懂牙科设备、懂采购节奏、还能帮你整理数据和解释推荐理由的智能助手。

你可以用它做什么

  • 导入和维护牙科设备/耗材/配件商品数据
  • 导入客户购买记录，沉淀真实采购行为
  • 自动生成用户画像，识别偏好、价值等级与补货节奏
  • 基于多路召回与 LLM 排序生成推荐结果
  • 通过 Agent 对话查询信息、解释推荐、辅助执行操作
  • 对推荐结果做采纳/拒绝反馈，帮助系统持续优化

第一次使用前请牢记

  • 系统的效果高度依赖数据质量，尤其是 SKU、客户 ID、购买日期。
  • 建议先完成“系统设置”，再导入数据。
  • 推荐结果不是一次点击就神奇出现，它依赖前面的商品、购买、画像链路。
  • 如果页面提示“连接失败”或“模型为空”，先回到系统设置检查模型与 API 配置。

01. 首次上手流程

如果你今天是第一次打开系统，请不要乱跳页面。最顺的路线是“先连模型，再导数据，再算画像，再看推荐”。

建议阅读 5 分钟

先进入「系统设置」

填写 Base URL、API Key，然后点击 测试连接 与 获取模型列表。 成功后，分别选择： 推荐/对话模型、增强/批量模型、Embedding 模型，再保存配置。

导入商品数据

进入「商品管理 → 原始商品」，导入包含 SKU、商品名称、旧 SKU、状态等信息的商品 Excel。 这是系统理解“卖什么”的基础。

执行商品增强

进入「商品管理 → 增强商品」，点击 开始 LLM 增强。 系统会为商品补充分类、用途、耗材、配件、关键词等结构化信息。

构建关系图谱

在增强商品页面继续构建关系，或进入「关系图谱」查看结果。 这一步会让系统更擅长理解“哪些商品会一起买、互补、替代或配套”。

导入购买记录

进入「购买记录」，导入客户历史购买明细。系统会基于这些记录生成用户画像，并为推荐提供真实业务依据。

查看画像并生成推荐

导入完成后先到「用户画像」看客户标签和补货提醒，再到「推荐结果」为单个客户或全部客户生成推荐。

一句话记忆： 先让系统知道“卖什么”，再让系统知道“谁买了什么”，它才可能给出有业务意义的推荐。

02. 适合谁使用

不同角色进入系统时，关注重点不同。你可以对照自己的岗位，快速定位常用页面。

销售 / 客户经理

重点关注用户画像、推荐结果、Agent 对话。你最常用的是：查客户、看推荐理由、跟进补货提醒。

运营 / 数据管理员

重点关注商品管理、购买记录、系统设置。你最重要的工作是把数据导对、导全、导干净。

管理者 / 负责人

重点看推荐结果是否合理、画像是否可信、反馈是否被系统吸收，以及整体使用流程是否顺畅。

04. 系统设置：系统能不能用，先看这里

如果系统像发动机，这一页就是点火系统。模型没配好，后面所有智能能力都会受影响。

连接配置 最优先

  • Base URL：模型服务地址，例如 OpenAI 或兼容平台地址。
  • API Key：访问模型服务的密钥。
  • 测试连接：用于确认系统能否连通模型服务。
  • 获取模型列表：拉取当前账号实际可用模型。

模型选择 直接影响效果

  • 推荐/对话模型：负责 Agent 对话和推荐排序。
  • 增强/批量模型：负责商品增强、批量结构化处理。
  • Embedding 模型：负责向量表示与相关召回能力。
重要提醒： 请先点击“获取模型列表”，再选择模型。不要随便手填一个看起来像模型名的字符串，除非你非常确定该模型可用。

高级设置

包括 temperature、max_tokens、timeout。 默认情况下，大多数用户可以沿用系统建议值；只有在你已经熟悉模型行为时，才建议修改。

LangSmith 监控

如果你的团队需要追踪 LLM 调用链路、分析问题或做效果监控，可以在这里开启 LangSmith。 如果你只是日常业务使用，可以先不启用。

05. 商品管理：把“卖什么”讲清楚

推荐效果好不好，首先取决于商品主数据是不是干净、完整、可理解。

原始商品

维护 SKU、商品名称、旧 SKU、在售/停售状态等核心信息。这里是所有后续能力的底座。

  • 支持 Excel 导入
  • 支持手工新增、编辑、删除
  • 支持 SKU 与旧 SKU 映射维护

增强商品

通过 LLM 把原始商品描述增强成结构化知识，例如品牌、分类、使用场景、关键词、耗材、配件等。

  • 先导商品，再做增强
  • 增强后可进一步构建关系图谱
  • 增强结果会直接影响推荐质量

关系图谱

帮助你看清商品之间的连接关系，例如同系列、消耗品、配件、互补关系和共现采购关系。

  • 可筛选关系类型
  • 适合复核系统的“商品理解”是否符合业务常识
推荐的商品数据处理顺序： 先导入原始商品 → 检查 SKU 和状态 → 执行 LLM 增强 → 构建关系图谱 → 再进入客户和推荐环节。

06. 购买记录：系统真正“认识客户”的地方

如果商品数据告诉系统“你卖什么”，购买记录就是在告诉系统“客户真正买了什么、多久买一次、买多少”。

支持的操作

  • Excel 导入客户购买记录
  • 按客户 ID、日期范围搜索
  • 手工新增、编辑、删除单条记录
  • 按用户分组查看采购情况

你最应该关注的数据质量

  • 客户 ID 要稳定，不要同一客户多个写法。
  • SKU 要尽量和商品主数据对得上。
  • 购买日期 决定了活跃度、采购周期、补货提醒。
  • 数量 影响客户价值和采购强度判断。
实用建议： 每次导入购买记录后，最好去“用户画像”抽查几个客户，看看画像是否符合你的业务直觉。

07. 用户画像：把冷冰冰的购买记录变成可读的客户判断

画像不是给系统自己看的，而是给业务人员快速判断客户状态用的。

画像通常会告诉你什么

  • 客户最近是否活跃
  • 偏爱哪些品类、品牌、产品类型
  • 采购频率和可能的补货周期
  • 客户价值层级（高 / 中 / 低）
  • 需要重点关注的补货提醒

如何读懂画像

不要只盯着一个字段。最有价值的方式是把“最近活跃度 + 品类偏好 + 补货信号”一起看。 例如：某客户近期活跃、偏好口腔耗材、某类采购周期已临近，那么这类商品推荐通常更值得优先跟进。

08. 推荐结果：从“建议什么”走到“为什么建议”

这一页不是只给出一个 SKU 列表，而是要给出可被业务理解、解释、复核的推荐结果。

你可以怎么用

  • 为单个客户生成推荐
  • 批量为多个客户生成推荐
  • 查看推荐理由、置信度、来源
  • 对推荐做采纳 / 拒绝反馈

你应该如何复核

  • 先看这个客户是否真的有对应品类历史
  • 再看推荐理由是否能解释得通
  • 最后看推荐来源和置信度，判断是否值得给客户
请记住： 推荐结果是“辅助决策”，不是“无条件执行”。业务人员的判断仍然非常重要。

09. Agent 对话：最适合“我想直接问”的场景

如果你不想在多个页面来回切，Agent 对话页就是你的快捷入口。

它能帮你做什么

  • 查询商品信息、客户画像、购买记录
  • 解释推荐为什么这么排
  • 辅助添加或删除购买记录
  • 触发推荐生成等操作
  • 展示思考过程和工具调用过程

怎么提问更有效

  • 带上明确客户 ID，例如：帮我看 KH3734 最近该推荐什么
  • 带上 SKU 或商品名，例如：VZ001286 是什么商品，适合搭配什么
  • 如果要执行操作，尽量说清楚时间、数量、SKU、客户 ID
一个很好用的习惯： 先用 Agent 查询和解释，再去对应页面做复核，这样效率会很高。

10. Token 使用量预估：先算账，再开跑

这是非常重要的一节。很多用户不是系统用不好，而是在批量增强、批量推荐、长对话过程中， 没有提前预估 Token 消耗，结果 API 额度、余额或并发资源不够，中途被迫终止。

强烈建议上线前阅读
先说明白： 下面的数字是经验预估值，目的是帮助你判断“资源大概够不够”，不是账单精确值。 实际消耗会受到模型种类、商品名称长度、购买记录多少、候选数量、对话轮数、以及模型输出长度影响。

系统里真正会消耗 LLM / Embedding Token 的地方

  • 商品增强：每个商品 1 次结构化抽取 + 1 次 embedding
  • 推荐生成：每个用户通常 1 次查询 embedding + 1 次 LLM 排序
  • Agent 对话：每轮至少 1 次模型调用，带工具时往往是多次
  • 知识图谱自动构建：按品类分组做关系推断，每组 1 次 LLM 调用

哪些操作几乎不消耗 Token

  • 查看原始商品列表、购买记录、画像、推荐结果
  • 系统设置里的“获取模型列表”与“测试连接”
  • 关系图谱的查看、筛选、缩放
  • 普通数据库查询与页面分页

也就是说，真正需要预算的，是“增强”“推荐”“对话”“图谱推断”这几类智能操作。

功能 大致调用方式 单次预估 Token 100 次 / 100 条量级预估 说明
商品增强（单个商品） 1 次 LLM 抽取 + 1 次 embedding 600 ~ 1,100 6 万 ~ 11 万 最常见的批量消耗项，商品名越长、输出越详细，消耗越高。
推荐生成（单个用户） 1 次查询 embedding + 1 次 LLM 排序 800 ~ 1,800 8 万 ~ 18 万 会受最近购买记录条数、候选商品数量、推荐理由长度影响。
Agent 对话（简单问答） 1 次对话模型调用 400 ~ 1,400 4 万 ~ 14 万 适合问“这是什么商品”“这个客户什么情况”这类轻查询。
Agent 对话（带工具查询） 通常 2 次模型调用 + 1 次工具查询 800 ~ 2,500 8 万 ~ 25 万 例如“帮我查 KH3734 最近买了什么并解释推荐”。
Agent 对话（带写操作确认） 多轮模型调用 + 确认恢复 1,200 ~ 3,500 12 万 ~ 35 万 例如通过助理添加购买记录、删除记录、生成推荐。
知识图谱自动构建（每个品类分组） 1 次 LLM 关系推断 1,000 ~ 2,400 10 组约 1 万 ~ 2.4 万 通常不是按“每个商品”计，而是按“每个品类分组”计。

商品增强预算怎么估

当前系统的商品增强流程是：每个商品 1 次结构化抽取 + 1 次 embedding。 所以最简单的估算公式是：

总 Token ≈ 商品条数 × 600 ~ 1,100

  • 10 条商品：约 6,000 ~ 11,000
  • 100 条商品：约 60,000 ~ 110,000
  • 1,000 条商品：约 60 万 ~ 110 万

如果你一次要增强几千条商品，建议先抽样跑 20~50 条，确认效果和消耗，再全量执行。

推荐生成预算怎么估

推荐生成通常是：1 次查询 embedding + 1 次 LLM 排序。 所以可以粗略按下面公式估：

总 Token ≈ 用户数 × 800 ~ 1,800

  • 10 个用户：约 8,000 ~ 18,000
  • 100 个用户：约 8 万 ~ 18 万
  • 1,000 个用户：约 80 万 ~ 180 万

如果你准备做“全量客户批量推荐”，这个预算一定要提前看，否则很容易在中途停掉。

Agent 对话为什么有时特别费 Token

因为对话不是永远只打一轮模型。带工具的 ReAct 对话通常会经历： “理解问题 → 决定调用什么工具 → 读工具结果 → 再组织回复”。 如果再加上写操作确认、恢复执行，轮次还会继续增加。

  • 只问一句简单问题：一般最省
  • 查询客户画像/购买记录/商品信息：中等消耗
  • 让 Agent 帮你执行写操作：相对更贵
  • 让 Agent 生成推荐：会叠加“推荐生成”的预算

知识图谱自动构建怎么估

这部分不是按单条商品算，而是按品类分组算。系统会把同一品类的一批增强商品组合成一个 prompt， 推断它们之间的治疗流程关系。

  • 1 个品类组：约 1,000 ~ 2,400 Token
  • 5 个品类组：约 5,000 ~ 12,000 Token
  • 10 个品类组：约 1 万 ~ 2.4 万 Token
推荐的预算策略： 不要按“理论刚刚够”去准备额度，建议至少预留 1.5 倍到 2 倍缓冲。 比如你预估 10 万 Token，会更稳妥的做法是按 15 万到 20 万来准备。

场景 1：首次导入 100 条商品

建议预算：6 万 ~ 11 万 Token。
如果你还要顺手构建知识图谱，再额外预留 1 万 ~ 2 万 Token 会更稳。

场景 2：给 100 个客户批量生成推荐

建议预算：8 万 ~ 18 万 Token。
如果客户购买记录很多、推荐理由写得更细，建议按上限甚至 1.5 倍准备。

场景 3：业务同事连续用 Agent

如果一天大概进行 100 轮混合型对话，建议预留： 8 万 ~ 25 万 Token。
若经常通过 Agent 直接执行操作或生成推荐，预算要再往上加。

最实用的做法： 首次上线时，不要一上来就跑“1000 条商品增强”或“全客户批量推荐”。先用 20 条商品、10 个客户 做试运行，把实际消耗看出来，再放大规模。

11. 一条推荐的标准工作流

下面这套流程最适合新团队复制执行。照着走，能最大程度避免“系统没问题，但结果看不懂”的情况。

步骤 1：完成系统设置

确认连接成功、模型列表已获取、三类模型都选好，并且配置已经保存。

步骤 2：导入商品 Excel

先把商品底座建好。尤其要关注 SKU、旧 SKU、状态字段是否正确。

步骤 3：执行商品增强与图谱构建

这一步决定系统对商品的“理解深度”，也决定后面推荐理由是否自然、专业。

步骤 4：导入购买记录

把真实客户采购历史带进来，系统才能建立用户画像与补货节奏。

步骤 5：抽查用户画像

随机抽几个重要客户，看看价值等级、偏好和补货提醒是否符合常识。

步骤 6：生成推荐并复核

看推荐结果页面的理由、置信度、来源；必要时回到 Agent 对话做进一步解释。

步骤 7：记录反馈

对明显靠谱的推荐做采纳，对不合适的推荐做拒绝，帮助系统逐渐贴近你的真实业务偏好。

12. 常见问题

如果你遇到问题，先看这里。很多“系统问题”其实是配置没填全、数据没导全、或顺序没走对。

Q1：为什么我保存系统设置失败？

通常先检查：Base URL、API Key、三个模型、以及高级设置中的 temperature、max_tokens、timeout 是否都有值。

Q2：为什么推荐结果为空？

先确认是否已经导入购买记录；再确认商品增强是否执行过；最后查看该客户是否有足够历史数据。

Q3：为什么 Agent 回答得不够准？

通常有三类原因：模型配置不合适、底层数据不完整，或者提问过于模糊。请尽量带上客户 ID、SKU、日期等具体信息。

Q4：导入购买记录后该先看什么？

先看“用户画像”。如果画像都不合理，直接看推荐通常只会更困惑。

Q5：为什么某些商品看起来关系很奇怪？

先检查原始商品名称和分类是否合理，再看增强结果是否偏离业务语义。商品关系图谱本质上依赖前面的商品数据质量。

Q6：这套系统能完全替代人工判断吗？

不能。它最擅长的是“整理信息、发现线索、解释依据”，最终对客户是否推荐、何时推荐，仍建议由业务人员把关。

13. 最后给使用者的建议

如果你想让系统越用越顺，下面这几条经验非常值得坚持。

保持数据命名稳定

同一个客户、同一个 SKU，请尽量始终使用同一种写法。稳定的主键，是系统理解业务的基础。

先抽查，再批量

无论是商品增强还是推荐生成，都建议先抽样看几个结果，确认方向没偏，再做大批量操作。

形成反馈闭环

推荐被采纳还是被拒绝，最好都留下反馈。系统真正变“懂你”，靠的不是一次配置，而是持续反馈。

最重要的一句话： 把它当成一个懂业务的智能助手，而不是一个会自动替你做决定的黑盒子。你越愿意和它配合，它给你的价值就越大。

附：推荐阅读顺序

如果这是你第一次接触本系统，请按这个顺序操作： 系统设置 → 商品管理 → 购买记录 → 用户画像 → 推荐结果 → Agent 对话。 这样你会最容易理解系统逻辑，也最不容易在中途因为数据或配置问题卡住。
